# Draft AI Long-Term Improvement Plan

Status: proposal / living document
Scope: pick + ban recommendation systems and their validation harnesses
Goal: continual, measurable iteration toward a **smarter, more realistic, and more dynamic** drafting opponent.

This document is an analysis of the *current* draft/ban AI and a phased roadmap for improving it. It is written to be executed incrementally: each phase produces a shippable improvement gated behind a regression check, so quality never silently regresses.

---

## 1. Executive Summary

The draft AI is a mature, well-instrumented **greedy linear scorer**. It scores each candidate as a weighted sum of Bayesian-smoothed historical signals (base winrate, ally synergy, enemy counter, counter risk) plus role-fit and composition-fit terms, with phase-dependent weights. It beats a random baseline by a very large margin (native_full ≈ 91.9%, native_softmax ≈ 89.8%) and is certified against a holdout dataset (~76% winner-prediction accuracy).

Its ceiling, however, is bounded by four structural facts:

1. **It is greedy and one-step.** No robust search of the draft tree. The only lookahead variants are quarantined for causing >20% side bias.
2. **It is fully deterministic at the scoring layer.** The gameplay softmax is now exercised by the validation harness via `native_softmax` (Workstream 0.1), so shipped and measured behavior match.
3. **It cannot adapt.** Weights are hardcoded constants. There is no player modeling, no meta drift, no learning loop, and stats are loaded once from static CSVs.
4. **It optimizes proxy signals, not outcomes.** Pairwise historical winrates are a strong proxy, but the recommender never closes the loop against ground-truth simulated match results during selection.

The improvement plan is organized around five workstreams — **Selection Realism**, **Search Depth**, **Model/Feature Quality**, **Data & Learning Loop**, and **Validation Infrastructure** — feeding a continual-iteration flywheel. The single highest-leverage early investment is the **validation infrastructure** (statistical A/B gate + self-play Elo ladder), because without it every subsequent change is unmeasurable.

## 2. Current System Architecture

### 2.1 Two parallel C++ systems

There are two draft-related C++ subsystems, and they are frequently confused. This plan keeps them distinct:

| System | Namespace | Files | Role today |
|---|---|---|---|
| **Production recommender** | `sim::draft_ai` | `sim_draft_ai_recommender.{hpp,cpp}`, `sim_draft_ai_evaluator.{hpp,cpp}`, `sim_draft_ai_stats_database.{hpp,cpp}`, `sim_draft_ai_types.hpp`, `sim_draft_ai_config.{hpp,cpp}` | Actual pick/ban recommendations used by gameplay and strategies; loads and validates `stats_manifest.json` for provenance and supports a certified `snapshot_id` via config |
| **Winner predictor / analysis** | `sim::draft` | `sim_draft_recommender.{hpp,cpp}` | Whole-team win-probability prediction, ablation/analysis tooling, `predict_draft_winner`. Certified holdout model lives here |

Binding layer: `native/src/teamfight_simulation_core.cpp` exposes both to GDScript.

### 2.2 GDScript integration

- Bridge: `scripts/simulation/native_simulation_backend.gd`
  - Pick: `get_draft_ai_pick_recommendations(stats_dir, available, allies, enemies, max_results, draft_step, strategy, config_path)`
  - Ban: `get_draft_ai_ban_recommendations(stats_dir, available, allies, enemies, max_results, draft_step, acting_side, weight_overrides, strategy, config_path)`
  - Legacy breakdown: `get_draft_recommendations_with_breakdowns(allies, enemies, available, top_n, stats_dir, base_weight, synergy_weight, counter_weight, draft_position, early_pick_base_weight, late_pick_counter_weight, config_path)`
  - `config_path` is optional and defaults to `""`; when it points to a JSON containing `certification.stats_snapshot_id`, the recommendation fails loudly if the loaded stats snapshot does not match.
- Strategy wrappers: `scripts/tools/draft_strategy*.gd` (native, native_softmax, native_lookahead, random, base_power_only, certified, native_ablation, and lookahead variants)
  - Native strategies pass `res://model_stats/draft_ai_config.json` as the default config path and return an empty result with an error if the native backend is unavailable or returns no recommendations.
- Gameplay opponent: `scripts/app/simulation_viewer_base.gd` (`_try_enemy_draft_ai`)

### 2.3 Data pipeline

Production stats live in `res://model_stats/stats_output_100k/`, generated by `--generate-stats` (`generate_simulation_stats.gd` / `stats_simulation_csv_generator.gd`) from simulated matches. Files consumed:

- `summary_stats.csv` — per-champion base stats (win rate, pick/ban frequency, etc.)
- `combat_stats.csv` — per-champion base win rates
- `role_stats.csv` — per-champion role assignment win rates
- `role_combinations.csv` — full 5-role composition win rates (with partial-multiset aggregation at query time)
- `matchup_with.csv` — ally synergy pairs
- `matchup_vs.csv` — enemy counter pairs
- `stats_manifest.json` — provenance manifest with schema version, snapshot ID, catalog version, generator metadata, and hashes of the canonical CSV files

Stats are **static**: loaded once, never updated at runtime, never fed back from live matches. The loader validates the manifest hashes on load (with cache invalidation driven by the manifest and CSV mtimes), and a configured `certification.stats_snapshot_id` will fail loudly if the snapshot identity does not match.

## 3. Current Implementation Analysis

### 3.1 Pick scoring

Per-candidate components (`sim_draft_ai_evaluator.cpp`):

- `base_power` = centered base winrate
- `ally_synergy` = mean centered synergy vs each ally
- `enemy_counter_value` = mean centered counter (candidate beats enemy)
- `counter_risk` = mean centered counter (enemy beats candidate)
- `role_fit` (`sim_draft_ai_recommender.cpp`) = fixed step function on ally role counts: not-present `+0.050`, once `+0.020`, twice `-0.030`, 3+ `-0.060`
- `comp_fit` = centered value of partial role-combination lookup

Final: `total_score = 0.35·base + 0.25·synergy + 0.25·counter − 0.15·risk + 0.10·role_fit + 0.08·comp_fit` (default phase).

Phase-aware weights (by `draft_step`):

| Phase (steps) | base | synergy | counter | risk | role_fit | comp_fit |
|---|---|---|---|---|---|---|
| Early pick (6–8) | 0.45 | 0.20 | 0.15 | −0.20 | 0.05 | 0.04 |
| Mid pick (9–11) / default | 0.35 | 0.25 | 0.25 | −0.15 | 0.10 | 0.08 |
| Late pick (16–19) | 0.25 | 0.30 | 0.35 | −0.10 | 0.15 | 0.10 |

Selection: sort by score desc, tie-break by champion name. **Top-1, deterministic.**

### 3.2 Ban scoring

Components: `enemy_pick_value` (candidate's value *for the enemy*), `enemy_synergy`, `counters_my_team`, `denial_value` = `max(0, enemy_pick_score) − max(0, own_pick_score)` (differential, positive-gated), `fills_enemy_role_need`, `enemy_comp_fit`.

Phase weights:

| Phase (steps) | denial | enemy_synergy | counters_my_team | fills_role | comp_fit |
|---|---|---|---|---|---|
| Phase 1 ban (0–5) | 0.75 | 0.15 | 0.20 | 0.05 | 0.04 |
| Default | 0.60 | 0.25 | 0.25 | 0.10 | 0.08 |
| Phase 2 ban (12–15) | 0.50 | 0.30 | 0.35 | 0.15 | 0.10 |

Side-aware early-ban fallback: when `denial_value` is negligible early, if the *enemy* gets first pick after the ban phase, add `early_fallback_multiplier(0.30) · max(0, enemy_pick_value)`; if *we* pick first, add 0 (don't ban our own targets). Runtime overridable via `weight_overrides`. Selection is again **top-1 deterministic**.

### 3.3 Bayesian smoothing (`sim_draft_ai_stats_database.cpp`)

```
adjusted = (wins + prior_winrate·prior_samples) / (samples + prior_samples)   // prior 0.5, 100 pseudo-samples
confidence = samples / (samples + confidence_prior_samples)                   // 100 prior
centered = (adjusted − 0.5) · confidence                                      // missing data → 0.0
```

Confidence shrinks low-sample signals toward neutral — good. But note: `confidence` is computed and then **only used to shrink the point estimate**; it is never surfaced to the selection layer for risk-aware decisions.

### 3.4 Lookahead (quarantined)

`NATIVE_LOOKAHEAD*` strategies do 1-ply search over the top-8 candidates, adjusting scores by the opponent's best deterministic response (`response_weight=0.35` picks, `denied_enemy_pick_weight=0.50` bans). They are **quarantined** because they introduced >20% (up to 42%) side bias in self-play. Root cause is almost certainly that the search assumes a deterministic top-1 opponent and is applied asymmetrically across the fixed snake order, amplifying the structural Blue-side advantage.

### 3.5 Determinism vs. gameplay stochasticity (RESOLVED)

- The C++ recommender is **fully deterministic** (no RNG).
- The **test strategy** `draft_strategy_native.gd` selects `recommendations[0]` — deterministic top-1.
- **Gameplay** `simulation_viewer_base.gd::_try_enemy_draft_ai` requests top-5 and applies `_softmax_select(recs, temperature=0.5, scale=100)` — stochastic.
- **Workstream 0.1 fix:** `DraftPolicy` (`scripts/tools/draft_policy.gd`) centralizes the softmax logic; both gameplay and the `native_softmax` validation strategy (`scripts/tools/draft_strategy_native_softmax.gd`) use the same temperature, top-k, and score scaling.

Consequence: **the behavior that ships to players is now measured by the validation harness.** The `native_softmax` numbers describe the actual policy players face. `native_full` remains available as a deterministic baseline for comparison.

### 3.6 Magic numbers

All weights, phase step ranges, role-fit steps, lookahead constants, priors, and the softmax temperature/scale are hardcoded literals scattered across C++ and GDScript. Per project rules (`AGENTS.md`: "No magic numbers"), these should be centralized into a single tunable config surface before any systematic tuning begins.

## 4. Strengths (preserve these)

- **Strong, certified proxy model.** Pairwise winrate signal is genuinely predictive (~80% test with the corrected pairwise logistic; ~76% certified holdout). The bottleneck is *use* of signal, not signal absence.
- **Good statistical hygiene.** Bayesian smoothing + confidence weighting handles sparse pairs correctly.
- **Rich per-decision instrumentation.** Every pick/ban emits a full component breakdown (`DraftPickScoreBreakdown` / `DraftBanScoreBreakdown`), enabling ablation and audits.
- **Phase awareness** already models early-vs-late draft priorities.
- **Ground-truth-based comparison exists.** The validation harness simulates the drafted teams to decide wins rather than trusting the predictor (avoids the obvious circularity trap for strategy comparison).
- **Side-bias is measured and understood** (structural ~19.6% Blue advantage in native self-play; ~2.6% for random, confirming it is strategy-specific, not engine bias).

## 5. Weaknesses & Limitations

Grouped by theme, roughly by impact.

### 5.1 Policy realism / dynamism
- Scoring layer remains deterministic, but the shipped softmax policy is now exercised by the validation harness via `native_softmax`. Difficulty tiers and personas are still future work.
- No difficulty tiers. A single policy serves all players; cannot present an "easy/normal/hard" opponent.
- No player modeling: the AI never reacts to *this* opponent's tendencies, comfort picks, or ban patterns.
- No bluffing, baiting, flex-pick ambiguity, or "save this pick" behaviors that make a human opponent feel alive.

### 5.2 Strategic depth
- Greedy one-step; robust lookahead quarantined.
- No counter-pick-chain reasoning (A>B>C), no "protect the carry" combo detection, no explicit win-condition modeling.
- Role model is a coarse step function; no lane/position assignment, no flex/multi-role handling.
- Picks evaluated independently — no pick-order optimization or trade-off between "take now vs. wheel back."

### 5.3 Model / feature quality
- Linear weighted sum cannot represent interactions (e.g., synergy that only matters given a specific enemy). Interaction terms exist only in the separate `sim::draft` predictor, not the production recommender.
- Confidence is computed but discarded at selection time — no risk-aware or exploration-aware behavior.
- Composition modeling limited to role fingerprints; documented experiments with mechanical signals and archetypes overfit and were archived.
- **Workstream 0.2 correction:** `EvaluatorWeights` in `sim_draft_ai_config.hpp` are **not** dead code. `DraftEvaluator::evaluate_candidate_pick/ban` compute a `total_score` using those weights, and `evaluate_candidate_ban` uses that intermediate pick `total_score` directly to derive `denial_value`. Changing the evaluator weights therefore affects ban recommendations. Only the final `total_score` at the end of `evaluate_candidate_ban` is overwritten by `DraftRecommender` with phase-aware weights.

### 5.4 Data & adaptation
- Static CSVs, generated offline, never updated from live/self-play results.
- No mechanism to detect or respond to meta shifts after a balance change; stale stats silently mislead.
- No provenance/versioning tying a recommender build to the exact stats snapshot it was certified on.

### 5.5 Validation gaps
- ~~**Deterministic-vs-softmax mismatch**~~ (resolved by Workstream 0.1).
- ~~No statistical significance: winrates reported without confidence intervals or hypothesis tests.~~ (resolved by E.1: Wilson CIs + two-proportion z-test in `native_draft_validation_analyzer.gd`).
- ~~No automated regression gate on draft quality~~ (resolved by E.2: `native_draft_quantitative_gate.gd` checks win-rate floors and side-bias ceilings).
- ~~No self-play Elo/ladder to rank strategy *strength* transitively.~~ (resolved by E.3: `native_draft_elo_ladder.gd` + `native_draft_elo_gate.gd`).
- No calibration tracking over time; no longitudinal metric history across model versions.
- Ban phase under-tested; several non-native strategies still ban randomly (explicit TODOs).
- Audit test states are hand-crafted and few (7–9 states).

## 7. Guiding Principles for Iteration

1. **Measure the policy you ship.** Validation must exercise the exact selection policy (temperature, difficulty tier) used in gameplay.
2. **No change without a gate.** Every improvement lands behind an automated, statistically-grounded regression check.
3. **Ground truth over proxy.** Prefer simulated match outcomes to predictor self-agreement whenever feasible.
4. **Config over constants.** All tunables live in one versioned surface; the code reads them.
5. **Small, reversible steps.** Each phase is shippable and independently revertible (aligns with `AGENTS.md` change discipline).
6. **Realism is a first-class objective**, co-equal with strength. A stronger bot that feels robotic fails the end goal.

## 8. Improvement Roadmap (Phased)

Each workstream is independently valuable; phases are ordered so infrastructure precedes the changes it must guard.

### Workstream 0 — Foundations (do first)

**0.1 Unify the selection policy.** ✅
Implemented. Introduced `DraftPolicy` (`scripts/tools/draft_policy.gd`) and a `native_softmax` validation strategy (`scripts/tools/draft_strategy_native_softmax.gd`) that share the same temperature/top-5 softmax selection as gameplay (`scripts/app/simulation_viewer_base.gd`). The validation harness now runs the exact shipped policy; the deterministic-vs-softmax mismatch (Section 3.5) is resolved. Difficulty tiers and personas remain future work under Workstreams A.1 and A.4.

**0.2 Centralize tunables.** ✅
Implemented. `sim::draft_ai::Config` (`native/src/simulation/sim_draft_ai_config.{hpp,cpp}`) holds pick/ban phase weights, phase step ranges, role-fit/fills-role step functions, lookahead constants, and evaluator weights, with default member initializers reproducing prior hardcoded behavior exactly. `DraftEvaluator`/`DraftRecommender` take an optional `const Config *` (defaults to the hardcoded struct). `get_draft_ai_pick_recommendations`/`get_draft_ai_ban_recommendations` accept an optional `config_path` that loads/caches a JSON override (missing/malformed file falls back to defaults with a warning; no recompile needed to tune values, only to add new config keys). GDScript softmax tunables (temperature/scale/top-k) are centralized in `scripts/tools/draft_ai_config.gd`, read by `simulation_viewer_base.gd` and `draft_strategy_native_softmax.gd`, sharing the same optional override file (`res://model_stats/draft_ai_config.json`, absent by default). Verified: native rebuild clean, `--check-only` passes, `--check-native-load`, `--check-native-simulation-tests`, `--check-match-telemetry`, fixture parity, `--check-draft-ui`, and `--check-benchmark` all pass.
Notes:
- `EvaluatorWeights` are **not** dead code: the intermediate pick `total_score` produced by those weights feeds `denial_value` in `evaluate_candidate_ban`, so changing them affects ban recommendations.
- Config caching now tracks the file modification time, so edits to `draft_ai_config.json` take effect without restarting Godot.
- The native JSON loader (`sim_draft_ai_config.cpp`) and GDScript loader (`scripts/tools/draft_ai_config.gd`) now validate that override values are numeric, finite, and (for int fields) within range, emitting a warning and falling back to defaults for non-numeric, infinite, or out-of-range values.

**0.3 Stats provenance/versioning.** ✅
Implemented. Every stats snapshot now includes `stats_manifest.json` (schema version 1) with `snapshot_id`, `generated_at`, `catalog_version`, `generator_name`, `generator_args`, and a `files` map of CSV hashes. `scripts/tools/stats_manifest.gd` reads, writes, and validates manifests; `scripts/tools/stats_simulation_csv_generator.gd` emits the manifest after all CSV files are written. The manifest only includes canonical CSV files (`summary_stats.csv`, `combat_stats.csv`, `role_stats.csv`, `role_combinations.csv`, `matchup_vs.csv`, `matchup_with.csv`), and the `snapshot_id` is a stable hash of the catalog fingerprint plus the sorted file hashes. The native `DraftStatsDatabase` (`native/src/simulation/sim_draft_ai_stats_database.{hpp,cpp}`) reads the manifest, validates every listed CSV file hash (as `String.hash()` 32-bit unsigned values via `String::num_uint64`), validates that `files` is a Dictionary, each hash is a String, and `snapshot_id` is a String, and warns when the manifest is missing, has an unsupported schema, contains invalid types, or has a hash mismatch, exposing `snapshot_id()` and `manifest_error()` for callers. The draft AI config (`sim::draft_ai::Config`) supports a `certification.stats_snapshot_id` field loaded from `model_stats/draft_ai_config.json`; when set, the native pick/ban recommendation methods (`get_draft_ai_pick_recommendations`/`get_draft_ai_ban_recommendations`) compare the loaded snapshot ID against the configured ID and return an empty result with an error if they differ (including when a hash mismatch is present). The legacy `get_draft_recommendations_with_breakdowns` API also accepts an optional `config_path` so the validation harness baseline is gated. The deterministic and softmax native strategies (`draft_strategy_native.gd`, `draft_strategy_native_softmax.gd`, `draft_strategy_native_lookahead.gd`) now pass the shared default config path to the native API and return an empty result with an error when the native backend is unavailable or the recommendation list is empty. The draft AI thread cache invalidates when the manifest or any canonical CSV file mtime changes, so hash mismatches are detected after edits.

Notes:
- `generator_args` records the CLI arguments passed to the generator; avoid passing secrets in stats-generation commands.
- `champion_catalog.gd` warns if a champion has no role when computing the catalog fingerprint.
- `validate_pick_recommendations.gd` extracts `--config-path` and passes it to the baseline oracle so the pick validation gate inherits certification checks.

Verified: native rebuild clean; `--check-only`, `--check-native-load`, `--check-native-simulation-tests`, `--check-match-telemetry`, `--check-draft-ui`, fixture parity, and `--check-stats-csv-determinism --skip-fallback-compare` all pass. A dedicated test script confirmed: no config succeeds; matching snapshot ID succeeds; mismatching snapshot ID returns empty; hash mismatch detected after cache invalidation; legacy API respects config certification.

*Exit gate:* harness reproduces gameplay policy; all constants read from config; existing baseline numbers reproduced within noise.

### Workstream A — Selection Realism & Dynamism

**A.1 Difficulty tiers via temperature schedule.** Expose Easy/Normal/Hard as temperature + top-k presets. Easy = high temperature (more mistakes/variety), Hard = low temperature (near-optimal). Validate each tier's win rate vs. baseline to calibrate perceived difficulty.

**A.2 Phase-varying temperature.** Lower temperature on high-stakes picks (last pick, counter windows), higher on low-stakes early picks — mirrors how humans commit hard only when it matters.

**A.3 Opponent (player) modeling.** Track the human's pick/ban frequencies and comfort picks across a session/match history; bias bans toward the player's high-frequency/high-performance champions and bias the AI away from mirror-matching. Start with a simple frequency prior; keep it explainable.

**A.4 Persona/archetype policies.** Optional draft "personalities" (aggressive counter-picker, greedy scaling, comfort-one-trick) implemented as weight overrides on the shared config. Adds variety and replayability without new model architecture.

*Gate:* realism metrics (pick entropy, unique-champion diversity across N drafts, human-likeness proxy) improve without strength dropping below tier target.

### Workstream B — Search Depth (fix and revive lookahead)

**B.1 Diagnose lookahead side-bias.** Instrument the quarantined lookahead to isolate why it amplifies Blue advantage. Hypothesis: asymmetric application across snake order + assuming a deterministic opponent.

**B.2 Symmetric, policy-aware lookahead.** Re-implement 1-ply search so the modeled opponent uses the *same stochastic policy* the AI uses (expectation over the opponent's softmax, not its argmax). Enforce side symmetry by construction; add a self-play side-bias test as a hard gate (must stay within the accepted structural band).

**B.3 Bounded search / flat-MC rollouts.** Where affordable, replace hand-tuned lookahead adjustment with short simulated rollouts to the end of draft using the current policy, scored by the certified winner predictor (fast) or by actual match simulation (slow, offline). This is the bridge toward MCTS.

**B.4 (Stretch) MCTS over the draft tree.** Full search with the winner predictor as leaf evaluator and the stochastic policy as the tree policy. Gate on both strength (Elo) and latency budget (draft decisions must stay well within interactive time).

*Gate:* strength improves in self-play Elo AND side-bias stays within the accepted band AND per-decision latency under budget.

### Workstream C — Model & Feature Quality

**C.1 Risk-aware selection.** Surface `confidence`/variance to the selection layer; allow tiers/personas to reward or penalize high-variance picks (ceiling-chasing vs. safe).

**C.2 Learned scorer to replace/augment the linear sum.** Train a small model (gradient-boosted trees or a compact MLP, exported to a form the native layer can evaluate) on (draft-state → win) data from simulated matches. Must clear a strict wiring gate vs. the current linear model on a holdout (the project already uses a "+2pp" style gate; reuse it). Keep the linear model as the explainable fallback.

**C.3 Interaction features in production.** Port the interaction terms (base×synergy, synergy×counter) that already exist in `sim::draft` into the production recommender, guarded by the same gate.

**C.4 Better composition modeling.** Revisit archetypes with regularization / more data now that the self-play loop (Workstream D) can generate targeted samples for sparse comps — the prior archetype experiments failed largely on overfitting small samples.

*Gate:* holdout accuracy/calibration improves past the wiring threshold; explainability breakdown preserved for audits.

### Workstream D — Data & Continual Learning Loop

**D.1 Self-play data generation.** Nightly/CI job: current policy drafts against itself and prior versions, plays out simulated matches, and writes fresh `combat/matchup/role_combination` stats plus (draft-state → outcome) training rows. This is the engine of continual improvement.

**D.2 Automated stats regeneration + certification.** On catalog/balance change, regenerate stats, re-certify against holdout, and block promotion if calibration regresses. Prevents stale-stats decay.

**D.3 Iterated-best-response training.** Periodically retrain the learned scorer (C.2) on self-play data, then re-enter the ladder (Workstream E). Track for convergence and for degenerate/exploitative equilibria.

*Gate:* regenerated stats reproduce or beat prior certification; ladder shows monotone (non-regressing) strength across versions.

### Workstream E — Validation Infrastructure (highest early leverage)

## 9. The Continual-Iteration Flywheel

Once Workstreams 0, D, and E exist, improvement becomes a loop rather than a series of one-off experiments:

```
        ┌────────────────────────────────────────────────┐
        │                                                  │
        ▼                                                  │
  1. Self-play + sim  ──►  2. Regenerate stats / train    │
     (Workstream D)             scorer (C, D)              │
        ▲                          │                       │
        │                          ▼                       │
  5. Promote / ship  ◄──  4. Regression gate + Elo ladder  │
     new best policy          A/B significance (E)         │
        ▲                          │                       │
        │                          ▼                       │
        └──────────  3. Candidate policy change  ──────────┘
                        (A / B / C workstreams)
```

Each turn of the flywheel: propose a change → generate/self-play data → evaluate against the ladder with significance → promote only if it clears the gate on **both strength and realism** → the new best becomes the opponent for the next round. Stale stats and silent regressions are structurally impossible because promotion requires passing the gate.

## 10. Metrics & Success Criteria

Track all of these per version; promotion requires no regression on the guarded set.

**Strength**
- Elo/Bradley–Terry rating from self-play ladder (primary).
- Win rate vs. fixed baselines (random, base_power_only, previous best) with 95% CI.
- Winner-prediction accuracy / calibration MSE vs. certified holdout (for the scorer).

**Fairness / correctness**
- Self-play side bias within accepted structural band.
- Draft validity (no duplicates/overlaps) — must stay perfect.
- Ablation sanity (removing a signal should not *improve* strength).

**Realism / dynamism**
- Pick/ban entropy and champion diversity across many drafts (higher = less robotic, within reason).
- Difficulty separation: measurable win-rate gap between Easy/Normal/Hard tiers.
- (If available) agreement with human drafts.

**Operational**
- Per-decision latency budget (interactive draft must stay snappy).
- Stats freshness/provenance check passes.

## 11. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Learned scorer overfits sparse comps (prior archetype/mechanical failures) | Strict wiring gate on holdout; regularization; use self-play to fill sparse regions before trusting archetypes |
| Reviving lookahead re-introduces side bias | Symmetric, policy-aware search + hard side-bias gate before any promotion |
| Evaluation circularity (predictor grades its own drafts) | Keep ground-truth *simulated matches* as the arbiter for strength; use predictor only as a fast leaf estimator, cross-checked periodically against sim |
| Stronger bot feels robotic | Realism metrics are co-equal promotion criteria; difficulty tiers + personas + temperature schedules |
| Self-play converges to a narrow/exploitable equilibrium | Ladder includes diverse fixed opponents and prior versions; monitor diversity metrics |
| Tuning churn destabilizes balance | All tunables versioned; changes land as reversible config diffs behind the gate |
| Latency blowup from search | Explicit per-decision time budget in the gate; predictor-as-leaf before full sim rollouts |
